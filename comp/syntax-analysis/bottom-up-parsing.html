Bottom-Up Parsing

自底向上语法分析 (Bottom-Up Parsing) 从叶子开始，通过不断向上归约构造语法树。自底向上语法分析等价于找到输入符号串的一个最左归约。

Reductions​

归约 (Reduction) 是推导的逆过程。

对于文法 GGG，如果有推导 α⇒∗β\alpha \xRightarrow{*} \betaα∗​β，则称 β\betaβ 可归约为 α\alphaα，或者 α\alphaα 是 β\betaβ 的一个归约。 特别的，如果有 α⇒β\alpha \Rightarrow \betaα⇒β，则称 β\betaβ 可直接归约为 α\alphaα，或者 α\alphaα 是 β\betaβ 的一个 直接归约 (Direct Reduction)。

  • 最左归约 (Leftmost Reduction)：最右推导的逆过程，也称为 规范归约 (Canonical Reduction)。
  • 最右归约 (Rightmost Reduction)：最左推导的逆过程。
Example

Handle​

对于文法 GGG，如果有 S⇒rm∗αS \xRightarrow[rm]{*} \alphaS∗rm​α (其中 SSS 是 GGG 的开始符号)，则称 α\alphaα 是 GGG 的一个 最右句型 (Right-Sentential Form)。对于 最左句型 (Left-Sentential Form)，我们有完全类似的定义。 前面说过，最右推导通常被称为规范推导，而最右句型也称 规范句型 (Canonical Sentential Form)。

对于文法 GGG，如果有 S⇒∗xAy⇒+xαyS \xRightarrow{*} xAy \xRightarrow{+} x \alpha yS∗​xAy+​xαy，则称 α\alphaα 是句型 xAyxAyxAy 关于 AAA 的 短语 (Phrase)。 特别的，如果 S⇒∗xAy⇒xαyS \xRightarrow{*} xAy \Rightarrow x \alpha yS∗​xAy⇒xαy，则称 α\alphaα 是 简单短语/直接短语 (Simple Phrase)。

一个句型的 最左简单短语 (leftmost simple phrase) 被称为这个句型的 句柄 (Handle)。

Note

如果仅仅有 A⇒+αA \xRightarrow{+} \alphaA+​α，并不能说明 α\alphaα 就是句型 xAyxAyxAy 关于 AAA 的一个短语，还必须要有 S⇒∗xAyS \xRightarrow{*} xAyS∗​xAy。

LR Parsers​

LR(k): Parsers of LR family.

  • L: Scan input from left to right.
  • R: Construct a rightmost derivation in reverse.
  • k: Use k input symbols of lookahead to make decisions.
    • k = 0 or 1 are of particular interests, k is assumed to be 1 when omitted.
Why LR Parsers?

LR(k) VS LL(k):

  • LR(k) is more powerful than LL(k).
    • LR parsers can be constructed to recognize virtually all programming-language constructs for which context-free grammars can be written. Non- LR context-free grammars exist, but these can generally be avoided for typical programming-language constructs.
    • The class of grammars that can be parsed using LR methods is a proper superset (真超集) of the class of grammars that can be parsed with predictive or LL methods. LL(k) ⊂\subset⊂ LR(k).
    • Eliminating left recursion or left factoring is not needed.
  • LR(k) is as efficient as LL(k).
    • Linear in time and space to length of input, same as LL(k).
  • LR(k) is as convenient as LL(k).
    • Can generate automatically from grammar (YACC, Bison).
An Ambiguous Grammar Can Never Be LR

考虑下面的 dangling-else 文法：

S→iEtS∣iEtSeSS \rightarrow i E t S \mid i E t S e SS→iEtS∣iEtSeS

如果我们有一个 shift-reduce parser 正处在下面的情况：

StackInput
$ ⋯iEtS\$ \ \cdots i E t S$ ⋯iEtSe⋯ $e \cdots \ \$e⋯ $

我们无法确定 iEtSiEtSiEtS 是否为一句柄，无论在它之下的栈内容是什么。此时，有两种可能的选择：

  • 将 iEtSiEtSiEtS 归约为 SSS。
  • 将 eee 移进，期待下一个 SSS，来完成 iEtSeSiEtSeSiEtSeS。

但我们不知道应该选择哪一个动作。因此，这个文法不是 LR(1) 的。

需要指出的是，LR 分析技术可作修改，来适用于一些二义性文法。 比如，上面的 dangling-else 文法也可用 LR 进行分析，我们可以规定当出现上面的情形时，将 eee 移进，而不是把 iCtSiCtSiCtS 归约为 SSS，这样符合程序语言的一般规定。

我们接下来要讨论的 LR 分析本质上都是寻找给定符号串的规范归约。 规范归约的关键问题是确定和寻找句柄。

LR(0)​

活前缀/可行前缀 (Viable Prefix) 指的是规范句型的一个前缀，这种前缀不含句柄之后的任何符号。

形式上，给定一个文法 GGG，如果存在一个具有下面形式的最右推导

S⇒rm∗αAω⇒rmαβ1β2ωS \xRightarrow[rm]{*} \alpha A \omega \xRightarrow[rm]{} \alpha \beta_1 \beta_2 \omegaS∗rm​αAωrm​αβ1​β2​ω

我们就称 γ=(NT∪T)∗\gamma = (NT \cup T)^{*}γ=(NT∪T)∗ 是 GGG 的一个活前缀，且 γ=αβ1\gamma = \alpha \beta_1γ=αβ1​。并称 A→β1⋅β2A \rightarrow \beta_1 \cdot \beta_2A→β1​⋅β2​ 是活前缀 γ\gammaγ 的 有效项 (Valid Item)。一个活前缀一定有有效项，一个有效项可能与多个活前缀相关联。

之所以称为活前缀，是因为在右边添加一些终结符之后，就可以使它成为一个规范句型。 在 shift-reduce parser 中，栈中符号串始终是最右句型的一个活前缀，它最多包含但不会越过当前句型的句柄 (当前句型指的是，栈中符号串和输入符号串构成的句型)。

Example

对于下面的最右推导：

S⇒bBa⇒bbAaS \Rightarrow bBa \Rightarrow bbAaS⇒bBa⇒bbAa

这里句柄是 bAbAbA，因此可行前缀有 bbb、bbbbbb、bbAbbAbbA，但不能是 bbAabbAabbAa，因为越过了句柄。

info

在 LR 分析过程中的任何时候，栈里的符号串 (自底向上) 应该构成活前缀，如果把输入串的剩余部分加在后边应成为规范句型。

tip

幸运的是，一个上下文无关文法的活前缀构成了一个正规语言，这说明我们可以用 DFA 来识别所有的活前缀。

LR 分析通过维护一系列状态，来跟踪当前的分析阶段，进而决定是移进还是归约。 一个状态实际上代表一个项目集。

文法 GGG 的一个 项目 (Item) 是 GGG 的一个产生式，产生式右部的某个位置有一个点 (dot)。项目通常也被称为 LR(0) 项目。

比如说，产生式 A→XYZA \rightarrow XYZA→XYZ 可形成 4 个项目：

tip

构造 LR(1) 项目规范族的方法与 LR(0) 基本相同，需要修改的是 CLOSURE\text{CLOSURE}CLOSURE 和 GOTO\text{GOTO}GOTO。

Computing CLOSURE
CLOSURE(I)
{
  repeat:
    for each item [A ➞ α‧Bβ, a] in I:
      for each production B ➞ γ in G':
        for each terminal b in FIRST(βa):
          add [B ➞ ‧γ, b] to set I;
  until: no more items are added to I;
  return I;
}
Computing GOTO
GOTO(I, X)
{
  initialize J to be the empty set;
  for each item [A ➞ α‧Xβ, a] in I:
    add item [A ➞ αX‧β, a] to set J;
  return CLOSURE(J);
}
Constructing of LR(1) Item Sets
ItemSets(G')
{
  initialize C to { CLOSURE( { [S' ➞ ‧S, $] } ) };
  repeat:
    for each set of items I in C:
      for each grammar symbol X:
        if GOTO(I, X) is not empty and not in C:
          add GOTO(I, X) to C;
  until: no new set of items are added to C;
}

Example

A grammar is LR(1) if and only if, for any state IiI_iIi​, the following two conditions are satisfied:

  • For any item [A→α⋅tβ,a]\lbrack A \rightarrow \alpha \cdot t \beta, a \rbrack[A→α⋅tβ,a] in IiI_iIi​ with ttt is a terminal, there is no item in IiI_iIi​ of the form [B→γ⋅,t]\lbrack B \rightarrow \gamma \cdot, t \rbrack[B→γ⋅,t]. (otherwise there is a shift-reduce conflict)
  • There are no two items in IiI_iIi​ of the form [A→α⋅,a]\lbrack A \rightarrow \alpha \cdot, a\rbrack[A→α⋅,a] and [B→β⋅,a]\lbrack B \rightarrow \beta \cdot, a \rbrack[B→β⋅,a]. (otherwise there is a reduce-reduce conflict)

LALR(1)​

LALR(1) 是 LR(1) 和 SLR(1) 的折衷。 LALR(1) 通过将 LR(1) 相似的状态合并来缩小生成的分析表。

LALR(1) 合并具有相同 核心 (Core) 的状态。

  • Core: LR(1) items minus the lookahead.
Example

合并状态可能会引入 reduce-reduce 冲突，但不会引入 shift-reduce 冲突。

  • shift-reduce conflicts are not introduced by merging:

    If state IijI_{ij}Iij​ is formed by merging IiI_iIi​ and IjI_jIj​, and IijI_{ij}Iij​ contains:

    • [A→α⋅,a]\lbrack A \rightarrow \alpha \cdot, a \rbrack[A→α⋅,a]
    • [B→β⋅aγ,b]\lbrack B \rightarrow \beta \cdot a \gamma, b \rbrack[B→β⋅aγ,b]

    Then, because the LR(1) item and LALR(1) item being merged have the same cores, either IiI_iIi​ or IjI_jIj​ must contain [A→α⋅,a]\lbrack A \rightarrow \alpha \cdot, a \rbrack[A→α⋅,a], and it must have an item [B→β⋅aγ,c][B \rightarrow \beta \cdot a \gamma, c][B→β⋅aγ,c] for some ccc. Well then this state has the same shift-reduce conflict, which means that this grammar is not LR(1).

  • reduce-reduce conflicts can be introduced by merging:

    For instance, state IiI_iIi​ contains:

    • [A→α⋅,a]\lbrack A \rightarrow \alpha \cdot, a \rbrack[A→α⋅,a]
    • [B→β⋅,b]\lbrack B \rightarrow \beta \cdot, b \rbrack[B→β⋅,b]

    while state IjI_jIj​ contains:

    • [A→α⋅,c]\lbrack A \rightarrow \alpha \cdot, c \rbrack[A→α⋅,c]
    • [B→β⋅,a]\lbrack B \rightarrow \beta \cdot, a \rbrack[B→β⋅,a]

    and the state IijI_{ij}Iij​ merged by IiI_iIi​ and IjI_jIj​ contains:

    • [A→α⋅,a/c]\lbrack A \rightarrow \alpha \cdot, a / c \rbrack[A→α⋅,a/c]
    • [B→β⋅,a/b]\lbrack B \rightarrow \beta \cdot, a / b \rbrack[B→β⋅,a/b]

    So when in state IijI_ijIi​j, if the lookahead symbol is aaa, there is an ambiguity in choosing which production to apply, either A→αA \rightarrow \alphaA→α or B→βB \rightarrow \betaB→β is OK. That is, there is a reduce-reduce conflict.

    A grammar is LALR(1) if and only if no conflicts are brought in merging LR(1) states.
